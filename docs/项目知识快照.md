# 项目知识快照

## 架构链路

项目核心链路为：文件扫描 -> 图片/PDF 读取 -> OCR -> 箱号候选提取 -> ISO 6346 校验 -> 分类输出 -> Excel 汇总。

## 当前模块

- `container_code.validator`：负责 ISO 6346 校验位计算和合法性判断。
- `container_code.extractor`：负责从 OCR 文本或文件名中提取合法箱号候选。
- `file_scanner`：负责递归扫描输入目录中的图片和 PDF。
- `image_loader`：负责把图片路径或 PDF 转图后的临时图片路径提供给 OCR。
- `ocr.base`：定义 OCR 引擎协议和识别结果结构。
- `ocr.tesseract_engine`：封装 Tesseract 调用，使用字符白名单和重试策略。
- `pipeline`：编排单文件识别流程，先用 OCR 文本提取箱号，再用文件名兜底。
- `output.classifier`：负责按识别状态复制文件到分类目录。
- `output.excel_writer`：负责生成 `识别结果.xlsx`，写入识别状态、来源、失败原因和耗时。
- `models`：集中定义跨模块传递的数据结构。

## 注意事项

- `HNKU6331795` 是当前核心验收样例。
- `HINKU6331795` 因结构不符合 `4 位大写字母 + 7 位数字`，不应被识别为合法箱号。
- OCR 与 PDF 相关第三方依赖使用延迟导入，避免未安装 Tesseract/Poppler 时影响核心规则测试。
- 后续 OCR 优化必须以样本集和基线数据为依据。
